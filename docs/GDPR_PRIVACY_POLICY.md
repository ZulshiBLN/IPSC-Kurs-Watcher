## 3. Legal Basis for Processing

IPSC Kurs Watcher processes personal data under **Article 6(1)(f) GDPR: Legitimate Interest**.

**Legitimate Interests:**
1. **Course Monitoring:** Enable users to receive notifications when courses become available
2. **Operational Logging:** Debug, troubleshoot, and maintain the application
3. **Deduplication:** Prevent duplicate notifications (improves user experience)

**Balancing Test:**
- Users explicitly configure email addresses and recipients (consent-like behavior)
- Necessary for core functionality (monitoring + notification)
- Minimal personal data collected (emails only, no tracking)
- Users can opt-out by removing from configuration
- No secondary use or data sharing

**Conclusion:** Legitimate interest outweighs privacy concerns given user control and minimal data collection.

## 4. Data Processing Activities

### 4.1 Email Addresses (Configuration)

**What is collected:**
- Sender email address (user's mailbox)
- Recipient email addresses (where to send notifications)

**How is it collected:**
- User manually enters in `config/config.json`
- Stored in plain JSON (user's machine, not uploaded)

**How is it used:**
- To send course availability notifications via Microsoft Graph API (email service)
- To identify which email account to use for sending

**How long is it kept:**
- Until user removes from configuration
- No automatic deletion

**Legal basis:** Legitimate interest (user-configured, necessary for notifications)

**Security:**
- Stored locally on user's machine
- Not shared with third parties except Microsoft (for email sending)
- Environment variables preferred over config file (see Security section)

### 4.2 Course Information (Monitoring)

**What is collected:**
- Course name, date, time, availability (number of slots), URL
- Scraped from shooting-store.ch website via HTML parsing

**How is it collected:**
- Automatic web scraping every 30 minutes (configurable)
- No personal data is collected from the website itself

**How is it used:**
- To determine if new courses are available
- To detect if availability decreased (alert user)
- To prevent duplicate alerts (deduplication via state file)

**How long is it kept:**
- Stored in `data/state.json` indefinitely
- Updated every monitoring cycle
- Can be manually deleted by user

**Legal basis:** Legitimate interest (necessary for functionality)

**Security:**
- Stored locally on user's machine only
- No transmission to external services except initial monitoring fetch

### 4.3 Operational Logs (System)

**What is collected:**
- Timestamps of monitoring events
- Application status messages (success, errors, retries)
- Counts (courses found, alerts sent, errors)
- HTTP response codes (for debugging)
- Sanitized error messages (sensitive data masked)

**How is it collected:**
- Automatically generated by the application
- Includes: date, time, component, message, context (JSON format)

**How is it used:**
- For application debugging and troubleshooting
- To understand why errors occurred
- To diagnose performance issues
- For security audit trail (critical endpoint access)

**How long is it kept:**
- 30 days (automatic cleanup)
- Older logs deleted automatically
- User can manually delete at any time

**Legal basis:** Legitimate interest (operational necessity)

**Security:**
- Email addresses are masked (***@***.***) in logs
- Passwords and API keys are masked ([REDACTED_SECRET])
- Logs stored locally on user's machine
- Not transmitted to external services

### 4.4 OAuth2 Tokens (Authentication)

**What is collected:**
- JWT access tokens issued by Microsoft Azure AD
- Token expiration time

**How is it collected:**
- Automatically obtained from Azure AD when user authenticates
- Stored in encrypted cache file

**How is it used:**
- To authenticate with Microsoft Graph API for email sending
- Short-lived (1 hour) and auto-refreshed

**How long is it kept:**
- 1 hour (auto-expires, then refreshed)
- Encrypted with DPAPI (Windows encryption)

**Legal basis:** Legitimate interest (necessary for email functionality)

**Security:**
- Encrypted with DPAPI LocalMachine scope
- Binary format (not human-readable)
- Never logged or transmitted unencrypted
- Auto-refreshed to minimize compromised token window

## 5. User Rights Under GDPR

Users have the following rights:

### 5.1 Right to Access (Article 15)

**What:** You can request to know what personal data is being processed.

**How:**
1. Open `config/config.json` – see your configured email addresses
2. Open `data/state.json` – see courses being tracked (no personal data)
3. Open `data/logs/*.log` files – see system logs (30 days of history)

**Timeline:** Immediate (direct file access, no request needed)

### 5.2 Right to Rectification (Article 16)

**What:** You can correct inaccurate personal data.

**How:**
1. Edit `config/config.json` and update email addresses
2. Save the file
3. Changes take effect on next application run

**Timeline:** Immediate

### 5.3 Right to Erasure (Article 17) – "Right to be Forgotten"

**What:** You can request deletion of your personal data.

**How:**
1. **Option A: Delete email addresses**
   - Edit `config/config.json`
   - Remove your email from the `recipients[]` array
   - Save file
   - You will no longer receive notifications

2. **Option B: Reset deduplication state**
   - Delete `data/state.json`
   - File will be recreated on next run
   - All previous courses will be treated as "new" (re-alerted)

3. **Option C: Delete logs**
   - Delete `data/logs/` directory
   - Logs are auto-generated; older ones deleted after 30 days anyway

4. **Option D: Complete cleanup**
   - Delete entire `data/` directory (state + logs)
   - Application will recreate with fresh state on next run

**Timeline:** Immediate

### 5.4 Right to Data Portability (Article 20)

**What:** You can request your data in a portable format.

**How:**
1. Copy `data/state.json` – all course data in JSON format
2. Copy `config/config.json` – all configuration in JSON format
3. Copy `data/logs/*.log` – all logs in JSON format (one entry per line)
4. All data is human-readable and machine-parseable

**Format:** JSON (open standard, not proprietary)

**Timeline:** Immediate

### 5.5 Right to Object (Article 21)

**What:** You can object to data processing.

**How:**
1. **Opt-out from notifications:** Remove your email from recipients list
2. **Disable monitoring:** Set `enabled: false` in monitor configuration
3. **Stop application:** Uninstall or disable Windows Scheduled Task

**Timeline:** Takes effect immediately

## 6. Data Retention Policy

| Data Type | Storage Location | Retention Period | Auto-Cleanup |
|-----------|------------------|------------------|--------------|
| **Email Addresses** | config/config.json | Indefinite (until manual delete) | No |
| **Course State** | data/state.json | Indefinite (until manual delete) | No |
| **System Logs** | data/logs/*.log | 30 days | Yes (automatic) |
| **OAuth2 Tokens** | data/.token_cache.json (encrypted) | 1 hour | Yes (auto-refresh) |

**Retention Rationale:**
- **Emails & State:** User-configured, necessary for functionality, user can delete anytime
- **Logs:** 30 days sufficient for debugging; GDPR data minimization principle
- **Tokens:** Minimized by auto-refresh (short-lived by design)

**Data Deletion Timeline:**
- User can delete any data immediately by deleting files
- Logs older than 30 days automatically deleted (without user action)
- OAuth2 tokens expire automatically after 1 hour

## 8. Data Security

### 8.1 Technical Safeguards

1. **Local Storage Only**
   - All data stored on user's personal machine
   - No cloud upload or synchronization
   - Filesystem encryption available (user responsibility)

2. **Credential Encryption**
   - OAuth2 tokens: DPAPI encrypted (Windows)
   - Secrets: Environment variables (not in config files)
   - Passwords: Never stored in plaintext

3. **Log Masking**
   - Email addresses masked: `***@***.***`
   - Credentials masked: `[REDACTED_SECRET]`
   - API keys masked: `[REDACTED_KEY]`

4. **Network Security**
   - HTTPS only (enforced for Azure AD, Graph API)
   - Certificate validation via Windows CA store
   - No insecure protocols (HTTP, FTP, etc.)

### 8.2 Organizational Safeguards

1. **Access Control**
   - Only user with machine access can view data
   - Scheduled Task runs as SYSTEM (Windows user)
   - File permissions: User's control

2. **Audit Logging**
   - Requests to critical endpoints (Azure AD, Graph) logged
   - Errors logged with context (for debugging)
   - Logs retained 30 days

3. **Code Review**
   - Open-source codebase (GitHub)
   - Subject to security audits
   - PSScriptAnalyzer linting enforced (security checks)

## 9. Breach Notification

**If unauthorized access to personal data occurs:**

### 9.1 Notification Timeline

1. **Upon Discovery:** Immediately assess severity
2. **Within 72 Hours:** Notify data protection authority
3. **Without Undue Delay:** Notify affected users (if high risk)

### 9.2 What We'll Do

1. **Contain:** Delete compromised tokens, secure access
2. **Document:** Record what happened, when, impact
3. **Investigate:** Determine root cause, prevent recurrence
4. **Report:** File with supervisory authority (DPA)
5. **Communicate:** Notify users of actions taken

### 9.3 Examples of Reportable Breaches

- Unauthorized access to token cache file
- Accidental public exposure of config file with credentials
- Malware access to log files containing email addresses

### 9.4 Your Role

If you discover a breach:
1. Stop the application immediately
2. Check `data/` directory for unauthorized changes
3. Contact your data protection authority
4. Contact Microsoft (if Azure/Graph breach suspected)

## 14. Appendix: Legal Basis Justification

### Article 6(1)(f) Legitimate Interest Analysis

**Legitimate Interests Pursued:**
1. Enable users to monitor course availability automatically
2. Maintain application reliability and security
3. Debug and troubleshoot operational issues

**Necessity:**
- Without email processing: Application cannot function (users wouldn't receive alerts)
- Without logs: Impossible to diagnose errors or improve reliability
- Without state files: Application would send duplicate alerts (poor UX)

**Proportionality:**
- Data collected: Minimal (emails, course data, logs)
- Data shared: None (kept locally)
- User control: Full (can delete at any time)
- Sensitivity: Low (no health, financial, or biometric data)

**Impact on Users:**
- Positive: Get course alerts automatically
- Negative: Email address stored (user-provided)
- Mitigation: Users control recipients list, can delete anytime

**Balancing:**
- User interests (get course alerts) > privacy impact (email stored locally)
- Legitimate interest clearly outweighs privacy concerns
- No less-intrusive alternative available

**Conclusion:** Article 6(1)(f) provides valid legal basis.
